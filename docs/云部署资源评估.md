# 波斯态势系统 · 互联网云服务器部署资源评估

> 基于当前代码架构（FastAPI + SQLite + Ollama 双模型 + React 静态前端）的务实评估。

---

## 一、资源需求总览

| 资源类型 | 最低配置（可跑通） | 推荐配置（体验接近本机） | 说明 |
|----------|--------------------|---------------------------|------|
| **CPU** | 4 核 | 8 核 | 爬虫并发、AI 推理（无 GPU 时吃满） |
| **内存** | 8 GB | 16–24 GB | 后端 + Ollama 双模型常驻或切换加载 |
| **GPU** | 无（仅 CPU 推理） | **8–10 GB 显存**（如 T4 / A10G） | 决定报告生成与新闻处理速度 |
| **系统盘** | 40 GB | 60 GB | 系统 + Python/Node 环境 + 模型文件 + SQLite |
| **数据盘** | 20 GB（可选） | 50 GB | 数据库与日志增长、备份 |
| **带宽** | 5 Mbps 出网 | 10+ Mbps | 爬取外网 RSS/API、用户访问 |
| **系统** | Ubuntu 22.04 LTS | 同左 | 便于安装 Docker / NVIDIA 驱动 / Ollama |

## 二、各组件资源占用

### 2.1 后端服务（FastAPI + Uvicorn）

- **CPU**：空闲时很低；抓取 18 个源时短时 1–2 核。
- **内存**：约 200–500 MB。
- **端口**：需开放 8100（API + WebSocket），或由 Nginx 反向代理。

### 2.2 Ollama + 双模型（主要变量）

| 模型 | 显存占用（GPU） | 内存占用（仅 CPU） | 单次推理耗时（参考） |
|------|------------------|---------------------|----------------------|
| qwen2.5:3b（新闻摘要/分类） | ~2 GB | ~4 GB | GPU 2–5 s，CPU 20–40 s |
| qwen3-vl:8b（战场报告 + 图像分析） | ~6 GB | ~10 GB | GPU 60–120 s，CPU 5–15 min |

- **当前逻辑**：调度器每 60 分钟跑一次抓取，约 30 分钟后再跑一次 `run_daily_analysis`；报告生成后调用 `unload_model()` 释放显存，因此**同一时刻通常只加载一个模型**。
- **显存峰值**：单模型最大约 **6 GB**（qwen3-vl:8b），建议 GPU **≥8 GB**（如 NVIDIA T4 16GB、A10G 24GB）。
- **无 GPU**：需 **16–24 GB 内存**，且报告生成会非常慢（数分钟级），易触发超时，仅适合“能跑就行”的演示。

### 2.3 前端

- 生产环境为 `npm run build` 后的静态资源（HTML/JS/CSS），由 Nginx 或后端挂载即可。
- **内存/CPU**：可忽略。

### 2.4 数据库与存储

- **SQLite**：单文件 `data/warfare.db`，当前规模约百 MB 级，年增长可按 500 MB–1 GB 预估。
- **模型文件**：Ollama 拉取后约 **qwen2.5:3b 2 GB + qwen3-vl:8b 6 GB ≈ 8 GB**。
- **系统 + 依赖**：约 5–10 GB。

**合计系统盘建议 40–60 GB。**

## 三、网络与安全

- **出站**：需访问 RSS、OpenSky/adsb.lol、YouTube 等；部分源在国外，建议云服务器**出网无大墙限制**（如海外区或具备良好国际出口）。
- **入站**：HTTP/HTTPS（如 80/443）、如需直接连 WebSocket 则保证与后端 8100 或 Nginx 代理可达。
- **防火墙**：仅开放 80/443（及 SSH 22），API 通过 Nginx 反向代理，不对外直露 8100。

## 四、云厂商与机型示例（按“推荐配置”）

以下为**带 GPU、体验接近本机**的选型思路（价格为参考，以控制台为准）：

| 云厂商 | 机型/规格 | GPU | 内存 | 说明 |
|--------|-----------|-----|------|------|
| **阿里云** | gn6i（T4） | 1×T4 16GB | 32 GB | 按量约 8–12 元/时，包月更划算 |
| **腾讯云** | GN7（T4） | 1×T4 16GB | 32 GB | 类似阿里云 |

## 五、部署架构示意

```
                    Internet
                        │
                        ▼
              ┌─────────────────┐
              │  Nginx (80/443)  │  ← SSL 终结、静态前端、/api 与 /ws 反向代理
              └────────┬─────────┘
                       │
         ┌─────────────┼─────────────┐
         ▼             ▼             ▼
   ┌──────────┐  ┌──────────┐  ┌──────────┐
   │ 静态前端  │  │ FastAPI  │  │  Ollama   │
   │ (Vite    │  │ :8100    │  │ :11434    │
   │  build)  │  │          │  │ qwen2.5   │
   └──────────┘  └────┬─────┘  │ qwen3-vl  │
                      │        └──────────┘
                      ▼
               ┌─────────────┐
               │ SQLite      │
               │ warfare.db  │
               └─────────────┘
```

- 单机部署：上述所有组件可放在同一台云主机。
- 若日后拆分：可将 Ollama 单独放到带 GPU 的机器，API 通过 `OLLAMA_BASE` 指向该机。

## 六、配置与改造要点

1. **Ollama 监听**：云上 Ollama 需监听 `0.0.0.0:11434` 或通过 Nginx 代理，且仅内网可达，不对外暴露 11434。
2. **数据库路径**：保持 SQLite 单文件即可；若多实例需共享，再考虑迁 PostgreSQL 等。
3. **环境变量**：`OLLAMA_BASE` 若指向本机可不变；若 Ollama 独立部署则改为 `http://<gpu-server>:11434`。
4. **前端 API 地址**：生产环境将请求发往当前域名（如 `https://your-domain.com/api`），通过 Nginx 代理到后端 8100，无需改前端代码（或构建时注入 `VITE_API_BASE`）。
5. **定时任务**：仍由本机 APScheduler 驱动，无需额外队列；若单机重启，需用 systemd/supervisor 拉起后端并保证 Ollama 常驻。

## 七、总结建议

| 场景 | 建议配置 |
|------|----------|
| **演示 / 个人使用** | 1 台 8 核 16 GB + **8 GB 显存 GPU**（如 T4），60 GB 系统盘，按量或包月。 |
| **无 GPU、能跑即可** | 1 台 8 核 24 GB 云主机，仅 CPU 推理；接受报告生成慢并调大超时。 |
| **小团队 / 轻度公网** | 同上 GPU 机，加 Nginx + HTTPS、防火墙只开 80/443/22，定期备份 SQLite。 |

**核心结论**：要获得与本机接近的体验，云上需要 **至少 8 GB 显存的 GPU** 和 **16 GB 以上内存**；CPU 仅作补充。存储与带宽按上述规格即可满足当前及一年内增长。
